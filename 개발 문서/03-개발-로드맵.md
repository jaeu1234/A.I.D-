## Phase C — 확장  🟡 진행 중

| 작업 | 수용 기준 | 상태 |
|------|-----------|------|
| 기기 간 동기화 (Firebase Firestore / Supabase) | A 기기에서 등록한 임시일정이 전자칠판에 자동 반영 | ✅ 완료 — Supabase(`overrides`/`ai_schedules` 테이블) + Realtime으로 임시일정·AI 시간표 동기화. 기본 시간표(`TEACHERS`)는 계속 코드에 하드코딩. |
| 서버사이드 인증 / 개별 PIN / 구글 로그인 | 클라이언트 PIN 노출 제거, 선생님별 계정 | ⬜ 미착수 |
| Claude API 서버 프록시 | API 키를 브라우저에서 제거(현재 upload.html은 키 노출 위험) | ⬜ 미착수 |
| 실제 학교 시간표 데이터 입력 | 샘플 6명 → 실제 교사·시간표로 교체 | ✅ 완료 — 1학년 1~10반 전체 반영(2026-07-08) |
| 반응형 / 전자칠판 풀스크린 최적화 | 대형 화면·터치 환경 UI | ⬜ 미착수 |

> **2026-07-08 Supabase 연동**: `src/lib/supabaseClient.js`(클라이언트) + `src/lib/sync.js`(메모리 캐시·CRUD·Realtime 구독) 추가.
> `location.js`의 `loadOverrides`/`loadAiSchedules`는 이제 이 캐시를 읽기만 하고, 실제 로드·쓰기·구독은 `sync.js`가 담당한다.
> `admin.html`/`upload.html`/`index.html`은 페이지 진입 시 `initSync()`를 호출해 초기 데이터를 불러오고, 다른 기기의 변경을
> Realtime으로 받아 현재 화면을 자동 갱신한다. 스키마는 `supabase_schema.sql` 참고.

## 현재 알려진 리스크
- **API 키 노출**: `upload.html`은 브라우저에서 Anthropic API를 직접 호출(키 localStorage 저장).
  실운영 전 반드시 Phase C의 서버 프록시로 이관.
- **PIN 취약**: `ADMIN_PIN` 하드코딩(`floors.js`). Phase C 인증으로 대체.
  Supabase RLS도 PIN과 동일한 수준(익명 키로 읽기/쓰기 전체 허용)이라 인증 이관 시 함께 강화 필요.
- **동기화 범위**: 임시일정·AI 시간표는 Supabase로 동기화되지만, 기본 시간표(`TEACHERS`)는 여전히 코드 배포로만 갱신된다.
